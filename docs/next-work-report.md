## 1. 현재 상태 요약

현재 저장소는 다음 범위까지는 실제 동작한다.

- 정적 프론트 빌드 가능
- nginx 단일 진입점 사용
- FastAPI + MySQL 로컬 스택 동작
- 네이버 로그인 연동
- 후기 / 댓글 / 이미지 업로드
- 현장 반경 기반 스탬프 적립
- 사용자 생성 경로와 좋아요
- 계정 분리 구조와 회원탈퇴 정리 규칙
- GitHub Actions CI 통과

즉, 화면만 있는 목업이 아니라 로컬 기준으로 흐름이 연결된 MVP다.

## 2. 지금 바로 해야 할 일

### 1순위. UI/UX 1차 정리
이유:
- 현재 가장 크게 체감되는 리스크는 기능 부족보다 화면 완성도다.
- PRD의 핵심인 `텍스트 최소화`, `스크롤 지양`, `아기자기한 탐색`이 아직 충분히 살아 있지 않다.

해야 할 일:
- 탐색 탭 상단 구조 정리
- 코스 탭 필터/카드 정렬 통일
- 스탬프 탭 상태 카드/버튼/거리 문구 통일
- 중복 상태 배너 제거

완료 기준:
- 390px 기준에서 줄바꿈, 카드 높이, 간격이 안정적일 것
- 상단 고정은 하단 탭 외 최소화될 것
- 상태 메시지가 다른 화면을 침범하지 않을 것

### 2순위. 후기 작성 정책을 스탬프와 연결
이유:
- PRD 방향상 실제 방문 기반 후기 구조가 중요하다.
- 어뷰징 방지와 서비스 신뢰도에도 직접 연결된다.

권장 정책:
- 해당 장소 스탬프 보유 사용자만 후기 작성 가능
- 스탬프 적립 후 24시간 이내 신규 후기 작성 허용
- 장소별 1일 1회 신규 작성 제한

완료 기준:
- UI에서 비활성 이유가 명확히 보일 것
- API와 DB 정책이 일치할 것
- 테스트가 추가될 것

### 3순위. 공공데이터 정식 연동
이유:
- 현재 장소 데이터는 번들 import 구조에 가깝다.
- 서비스 신뢰도와 운영 효율을 위해 실제 데이터 소스를 붙여야 한다.

해야 할 일:
- 대전 관광/행사 API 실제 연동
- 정규화 기준 확정
- 관리자 import 흐름 보강
- fallback 정책 정리

완료 기준:
- 실제 외부 데이터 1개 이상이 반영될 것
- 외부 응답이 변해도 앱 모델이 바로 깨지지 않을 것

### 4순위. 모바일 QA 체계화
이유:
- 지금까지 발생한 문제의 대부분이 레이아웃/상태 표현/모바일 UX였다.

해야 할 일:
- 화면별 QA 체크리스트 확정
- 390px 기준 검수 항목 문서화
- 로그인 / 후기 / 스탬프 / 관리자 시나리오 점검 항목 정리

### 5순위. 운영 구조 정리
이유:
- 기능은 늘어나고 있는데 운영 문서는 아직 뒤따라오지 못했다.

해야 할 일:
- 관리자 화면 분리 여부 결정
- 시크릿/환경변수/배포 경로 정리
- Supabase 실제 적용 시 쿼리와 버킷 절차 정리

`보이는 화면`, `실제 기록 정책`, `실데이터`, `운영 준비`를 PRD 방향에 맞게 조여 가는 것이다.

즉 지금은 기능 확장보다 품질 정렬 구간이다.
