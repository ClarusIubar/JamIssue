# 계정/삭제 규칙 정리

## 1. 내부 사용자 ID와 소셜 로그인 분리

현재 기준의 핵심 원칙은 `user.user_id`가 서비스 내부의 고유 계정 ID이고, 네이버/카카오 같은 외부 로그인 식별자는 `user_identity`에서 따로 관리한다는 점이다.

- `user`
  - 서비스 내부 계정 본체
  - 피드, 댓글, 스탬프, 사용자 경로는 모두 이 `user_id`만 참조한다.
- `user_identity`
  - `provider`
  - `provider_user_id`
  - `user_id`
  - 한 계정에 네이버와 카카오를 둘 다 연결할 수 있다.

이 구조의 장점은 다음과 같다.

- 네이버에서 카카오로 로그인 수단이 바뀌어도 내부 `user_id`가 유지된다.
- 후기, 스탬프, 추천 경로가 로그인 제공자 변경 때문에 끊기지 않는다.
- 인증 수단과 도메인 데이터가 분리되어 이후 확장이나 마이그레이션이 쉬워진다.

현재 기준에서는 `email`이 같아도 자동 연결하지 않는다.

- 네이버/카카오는 기본적으로 각각 별도 identity로 처리한다.
- 같은 이메일이라고 해도 자동 병합하지 않는다.
- 이미 로그인한 사용자가 명시적으로 계정 연결을 수행할 때만 하나의 내부 `user_id`에 묶는다.
- 이렇게 해야 잘못된 이메일 일치나 공유 메일, 공급자별 이메일 정책 차이로 인한 오연결을 막을 수 있다.

## 2. 삭제 규칙

### 회원 탈퇴

`DELETE /api/my/account`

회원 탈퇴는 내부 `user_id` 기준으로 정리한다.

삭제 대상:
- 사용자 identity
- 사용자가 작성한 후기(feed)
- 사용자가 누른 후기 좋아요(feed_like)
- 사용자가 작성한 댓글(user_comment)
- 사용자가 모은 스탬프(user_stamp)
- 사용자가 만든 추천 경로(user_route)
- 사용자가 누른 추천 경로 좋아요(user_route_like)

효과:
- 사용자의 피드/추천/좋아요/스탬프가 한 번에 정리된다.
- PRD의 사용자 기록 흐름은 유지하되, 탈퇴 시 흔적이 남지 않도록 설계했다.

### 후기 삭제

`DELETE /api/reviews/{review_id}`

후기를 삭제하면 그 후기 아래의 아래 데이터도 함께 사라진다.

- 댓글(user_comment)
- 후기 좋아요(feed_like)

즉 `feed`가 삭제되면 그 하위 상호작용도 같이 정리된다.

### 댓글 삭제

`DELETE /api/reviews/{review_id}/comments/{comment_id}`

댓글은 즉시 hard delete 하지 않고 `is_deleted=true`로 soft delete 한다.

이유:
- 대댓글 문맥을 유지할 수 있다.
- 사용자는 "삭제된 댓글입니다." 형태로 부모 댓글이 지워졌음을 알 수 있다.
- 답글 자체는 그대로 남아 사용자 대화 흐름이 무너지지 않는다.

추가 DB 규칙:
- `parent_id`는 self FK이며 `ON DELETE SET NULL`
- 그래서 회원 탈퇴처럼 댓글 행이 실제로 삭제되는 경우에도 자식 댓글은 남을 수 있다.

### 추천 경로 삭제

`DELETE /api/community-routes/{route_id}`

사용자가 만든 추천 경로를 직접 지우면:
- 경로 본체(user_route)
- 경로 장소(user_route_place)
- 경로 좋아요(user_route_like)
가 함께 정리된다.

## 3. PRD 관점에서 왜 유리한가

이 구조는 PRD의 다음 요구를 지키는 데 유리하다.

- 사용자 중심 기록 구조 유지
- 로그인 수단보다 서비스 내부 계정을 우선하는 구조
- 사용자 생성 경로 생태계 확장 가능
- 탈퇴/삭제 시 데이터 정리 기준이 명확함
- 댓글/대댓글 관계를 유지하면서도 피드 단위 정리는 깔끔함

즉, 이번 변경은 기능을 늘린 것이 아니라 PRD의 핵심인
`사용자 기록의 연속성`, `관계형 무결성`, `향후 확장성`을 지키기 위한 기반 정리다.
